## 三、新功能建议

### P0 - 核心必备

#### 1. 流式翻译显示 (Streaming)
- **现状**: 等整个批次翻译完成才显示，长文本等待体验差
- **建议**: 使用 SSE/stream 模式，逐字显示翻译结果
- **收益**: 显著降低用户感知等待时间，对标沉浸式翻译的核心体验

#### 2. 键盘快捷键
- **现状**: 只能通过 FAB 按钮或右键菜单触发
- **建议**:
  - `Alt+T` 翻译当前页面
  - `Alt+Y` 显示/隐藏翻译
  - `Alt+D` 翻译选中文本
  - 支持在 options 页自定义快捷键
- **收益**: 高效用户的刚需，所有竞品都支持

#### 3. 网站白名单/黑名单
- **现状**: 所有网站都注入内容脚本，无法排除或自动翻译
- **建议**:
  - 白名单: 打开即自动翻译的网站列表
  - 黑名单: 永不注入 FAB 的网站列表
  - 支持通配符匹配 (`*.github.com`)
- **收益**: 频繁阅读外文网站的用户体验提升巨大

#### 4. 鼠标悬停翻译
- **现状**: 只有右键划词翻译
- **建议**: 按住特定键（如 Alt）悬停段落时显示翻译 tooltip
- **收益**: 比右键菜单更快捷的交互方式

### P1 - 体验提升

#### 5. 输入框翻译
- **现状**: 只翻译页面静态文本，跳过 textarea/input
- **建议**: 在输入框旁添加翻译按钮，支持将输入内容翻译后回填
- **收益**: 写邮件、评论等场景的刚需

#### 6. 自定义翻译 Prompt 模板
- **现状**: Prompt 硬编码在各 provider 中
- **建议**: 在 options 页面提供 Prompt 编辑器，支持预设模板（直译/意译/学术/口语）
- **收益**: 不同场景下翻译风格可控

#### 7. 翻译历史记录
- **现状**: 缓存仅用于避免重复 API 调用，用户无法查看历史
- **建议**: 独立的翻译历史页面，支持搜索、导出、收藏
- **收益**: 语言学习场景的重要功能

#### 8. 自定义 API 地址 (完善)
- **现状**: popup.html 有 apiUrl 字段但默认隐藏，未实际生效
- **建议**: 每个 provider 支持自定义 endpoint，解决国内用户代理访问需求
- **收益**: 解决网络限制问题，支持私有部署的模型服务

#### 9. 多语言 UI
- **现状**: 插件界面全英文
- **建议**: 至少支持中英两种界面语言，跟随浏览器语言自动切换
- **收益**: 中文用户占比大，降低使用门槛

### P2 - 锦上添花

#### 10. 翻译质量对比
- **建议**: 选中文本后可同时用多个模型翻译，并排对比结果
- **收益**: 帮助用户选择最适合的模型

#### 11. 生词本/标注功能
- **建议**: 双击翻译后的单词可添加到生词本，支持导出为 Anki 卡片
- **收益**: 语言学习场景

#### 12. 翻译统计
- **建议**: 统计翻译字数、API 调用次数、费用估算
- **收益**: 帮助用户控制 API 开销

#### 13. 右键菜单增强
- **建议**: 右键菜单支持"翻译到指定语言"子菜单，无需进设置切换
- **收益**: 多语言用户的快捷操作

## 四、性能优化建议

### 1. Intersection Observer 按需翻译
- **现状**: 提取全页文本一次性翻译，长页面首次翻译慢且浪费 API 额度
- **建议**: 只翻译视口内及预加载区域的文本，滚动时动态翻译新内容
- **收益**: 长页面性能提升 3-10x，节省 API 费用

### 2. 并发批次翻译
- **现状**: 批次串行执行 (`for...of` 循环 await)
- **建议**: 控制并发数（如 3 个批次并行），用 Promise 池管理
- **收益**: 整页翻译速度提升 2-3x

### 3. 缓存策略优化
- **现状**: 单一 Chrome storage 缓存，1000 条上限
- **建议**:
  - 引入内存缓存层（Map），减少 storage 读取
  - 按网站分区存储，提高缓存命中率
  - IndexedDB 替代 chrome.storage 突破容量限制
- **收益**: 缓存命中率和读取速度提升

### 4. MutationObserver 动态内容翻译
- **现状**: 只翻译页面加载时的内容，SPA 路由切换或动态加载内容不会翻译
- **建议**: 监听 DOM 变化，自动翻译新插入的文本节点
- **收益**: 支持 SPA 应用和无限滚动页面

## 五、UI/UX 改进建议

### 1. Popup 设计优化
- 当前 3 tab 布局信息密度低，建议合并为单页滚动布局
- API Key 输入区应有更明确的状态指示（已配置/未配置/已过期）
- 每个 provider 显示 logo 图标增强辨识度

### 2. 翻译样式增强
- 支持用户自定义翻译文本颜色和背景色
- 支持选择翻译文本的字体
- 翻译标记支持折叠/展开

### 3. Onboarding 引导
- 首次安装时的设置引导流程
- 快捷键提示 overlay

### 4. 错误提示优化
- 区分错误类型给出具体指引（key 过期 vs 余额不足 vs 网络问题）
- 提供重试按钮而非只显示错误消息

## 六、技术债务

| 问题 | 建议 |
|------|------|
| LANGUAGE_NAMES 在每个 provider 中重复定义 | 提取到共享模块 `utils/languages.js` |
| `_parseBatchResponse()` 在每个 provider 中重复 | 移入基类 `TranslationProvider` |
| OpenAI 兼容 provider 代码高度相似 | 提取 `OpenAICompatibleProvider` 中间基类 |
| content.js 638 行单文件 | 拆分为 fab.js, translator-ui.js, dom-utils.js |
| 无构建工具 | 引入 Vite/webpack 支持打包、压缩、环境变量 |
| 无 TypeScript | 渐进迁移到 TS 提升类型安全 |
| 无 E2E 测试 | 引入 Puppeteer/Playwright 测试真实翻译流程 |
| 无 CI/CD | GitHub Actions 自动运行测试和打包 |
